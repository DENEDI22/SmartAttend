# Phase 3 — UI Design Contract

> Visual and interaction contract for the Admin Interface. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (Pico CSS classless framework) |
| Preset | not applicable |
| Component library | Pico CSS v2.1.1 (semantic HTML elements) |
| Icon library | none (use Unicode symbols and Pico CSS built-in indicators) |
| Font | system-ui stack (Pico default: system-ui, Segoe UI, Roboto, etc.) |

**Rationale:** Project uses FastAPI + Jinja2 with Pico CSS served from `static/`. No JS build step, no React, no shadcn. All UI is server-rendered HTML with Pico's classless semantic styling.

---

## Spacing Scale

Pico CSS manages spacing internally via `--pico-spacing: 1rem` (16px). The admin interface uses Pico's built-in spacing and overrides only where noted.

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Inline icon gaps (status dots) |
| sm | 8px | Compact table cell padding (Pico default) |
| md | 16px | Default `--pico-spacing` for forms and cards |
| lg | 24px | Section separation between CRUD areas |
| xl | 32px | Page top/bottom margin inside `<main>` |
| 2xl | 48px | Not used in this phase |
| 3xl | 64px | Not used in this phase |

Exceptions: Touch targets for enable/disable toggle buttons must be minimum 44px height for mobile usability.

---

## Typography

Pico CSS controls typography via CSS custom properties. The admin interface uses Pico defaults without override.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px (1rem, Pico default `--pico-font-size: 100%`) | 400 (`--pico-font-weight`) | 1.5 (`--pico-line-height`) |
| Label | 16px (Pico `<label>` default) | 400 | 1.5 |
| Heading | 20px (Pico `<h2>` for section titles) | 700 (Pico bold default for headings) | 1.2 (Pico heading default) |
| Page title | 28px (Pico `<h1>`) | 700 | 1.2 |

**Heading hierarchy for admin pages:**
- `<h1>` -- Page title (e.g. "Geraeteverwaltung")
- `<h2>` -- Not used within a single CRUD page
- `<hgroup>` -- Page title + subtitle description

## Color

Pico CSS provides a complete color system via CSS custom properties. The admin interface uses Pico's default light theme.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `var(--pico-background-color)` (#fff) | Page background, table background |
| Secondary (30%) | `var(--pico-card-background-color)` | `<article>` cards wrapping forms and tables |
| Accent (10%) | `var(--pico-primary)` (#1a6fb5) | Primary action buttons, active nav link |
| Destructive | `var(--pico-del-color)` (#c62828) | Deactivate user button, delete schedule button, error messages |

Accent reserved for:
- "Speichern" (Save) submit buttons on create forms
- "Hinzufuegen" (Add) buttons for new records
- Active navigation tab/link indicating current section

**Status indicators (inline, not accent):**
- Online: `color: #2e7d32` (green) -- small text or dot
- Offline: `color: var(--pico-muted-color)` (grey)
- Enabled: standard text
- Disabled/Deactivated: `color: var(--pico-muted-color)` with strikethrough or muted row

## Layout Contract

### Navigation

Admin pages use a `<nav>` element inside `<main class="container">` with three links:

```
Geraete | Benutzer | Stundenplan          [Abmelden]
```

- Navigation uses Pico's `<nav>` with `<ul>` pattern
- Active section indicated by `aria-current="page"`
- Logout button right-aligned using Pico's nav flex layout

### Page Structure

Each admin section follows this template:

```html
<main class="container">
  <nav>...</nav>
  <hgroup>
    <h1>{Section Title}</h1>
    <p>{Section description}</p>
  </hgroup>
  <!-- List/Table -->
  <figure><table>...</table></figure>
  <!-- Create Form (below table, inside <article>) -->
  <article>
    <h2>{Form heading}</h2>
    <form>...</form>
  </article>
</main>
```

### Devices Page (`/admin/devices`)

**Table columns:**

| Column | Source field | Format |
|--------|-------------|--------|
| Geraete-ID | `device_id` | Plain text |
| Raum | `room` | Plain text, editable inline or via form |
| Label | `label` | Plain text, editable inline or via form |
| Status | `is_online` | Green dot + "Online" / Grey dot + "Offline" |
| Aktiviert | `is_enabled` | Toggle button: "Aktivieren" / "Deaktivieren" |
| Zuletzt gesehen | `last_seen` | Relative time or datetime, or "--" if null |
| Lux | `last_lux` | Number with 1 decimal, or "--" if null |

**Device edit form:** Inline `<form>` per row for room/label assignment, or a dedicated edit form below the table. Each row has a POST form for enable/disable toggle.

### Users Page (`/admin/users`)

**Table columns:**

| Column | Source field | Format |
|--------|-------------|--------|
| Name | `first_name` + `last_name` | "Vorname Nachname" |
| E-Mail | `email` | Plain text |
| Rolle | `role` | Badge-style: "Admin", "Lehrer", "Schueler" |
| Klasse | `class_name` | Plain text, or "--" if null |
| Status | `is_active` | "Aktiv" / "Deaktiviert" |
| Aktion | -- | "Deaktivieren" button (destructive) |

**Create user form fields:**

| Field | HTML element | Validation |
|-------|-------------|------------|
| Vorname | `<input type="text" required>` | Required |
| Nachname | `<input type="text" required>` | Required |
| E-Mail | `<input type="email" required>` | Required, unique |
| Passwort | `<input type="password" required>` | Required, min 8 chars |
| Rolle | `<select required>` | Options: admin, teacher, student |
| Klasse | `<input type="text">` | Optional, shown when role=student |

### Schedule Page (`/admin/schedule`)

**Table columns:**

| Column | Source field | Format |
|--------|-------------|--------|
| Wochentag | `weekday` | German name: Mo, Di, Mi, Do, Fr, Sa, So |
| Startzeit | `start_time` | HH:MM |
| Endzeit | `end_time` | HH:MM |
| Geraet | `device_id` (FK) | Device label or device_id |
| Lehrer | `teacher_id` (FK) | Teacher full name |
| Klasse | `class_name` | Plain text |
| Aktion | -- | "Loeschen" button (destructive) |

**Table sorting:** Rows sorted by weekday (Mon-Sun), then by start_time ascending.

**Create schedule entry form fields:**

| Field | HTML element | Validation |
|-------|-------------|------------|
| Geraet | `<select required>` | Populated from devices list |
| Lehrer | `<select required>` | Populated from users with role=teacher |
| Klasse | `<input type="text" required>` | Required |
| Wochentag | `<select required>` | Mon-Sun (0-6) |
| Startzeit | `<input type="time" required>` | Required |
| Endzeit | `<input type="time" required>` | Required, must be after start |

## Interaction Patterns

### Form Submissions
- All forms use standard `POST` with page reload (no JavaScript required)
- Success: redirect back to the same admin section with a flash message
- Error: re-render the page with error message displayed using `<small role="alert" style="color: var(--pico-del-color)">` (established pattern from login.html)

### Destructive Actions
- Deactivate user: `<button>` with `class="secondary"` or `class="outline"` styled with `--pico-del-color`; uses browser `confirm()` dialog via `onclick="return confirm('...')"` (minimal JS, no build step)
- Delete schedule entry: Same pattern as deactivate user
- No hard deletes in this phase -- deactivate sets `is_active=False`, schedule delete is a true DELETE

### Device Enable/Disable Toggle
- Each device row contains a small `<form method="post">` with a single submit button
- Button text changes based on current state: "Aktivieren" (if disabled) / "Deaktivieren" (if enabled)
- Button uses Pico's `class="outline"` for secondary visual weight

### Flash Messages
- Rendered as `<p role="alert">` at the top of the content area, below `<hgroup>`
- Success: `style="color: #2e7d32"` (green)
- Error: `style="color: var(--pico-del-color)"` (red)
- Passed via query parameter `?msg=success&type=create` or template context variable

### Responsive Behavior
- Pico CSS handles responsive layout via `.container` max-width
- Tables wrap in `<figure>` for horizontal scroll on narrow screens (Pico built-in)
- Forms stack naturally as block-level elements
