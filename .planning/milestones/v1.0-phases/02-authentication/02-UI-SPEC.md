| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Inline icon gaps (not used this phase) |
| sm | 8px | Field-to-label gap, helper text offset |
| md | 16px | Default form field vertical spacing (Pico CSS `--form-element-spacing-vertical` default) |

## Typography

All type is rendered by Pico CSS defaults. No custom font is loaded. The following values match Pico CSS v2 defaults (system-ui stack).

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px | 400 | 1.5 |
| Label | 14px | 400 | 1.5 |
| Heading (h1 page title) | 28px | 700 | 1.2 |
| Small / helper / error | 14px | 400 | 1.4 |

Pico CSS controls all sizing via CSS custom properties. Do NOT add inline `font-size` declarations; use semantic HTML elements (`<h1>`, `<p>`, `<small>`) and let Pico CSS apply its scale.

## Color

Pico CSS provides a built-in light/dark color system. Phase 2 uses the **default Pico CSS theme** with no custom palette overrides. The 60/30/10 split maps to Pico CSS semantic variables:

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `--background-color` (Pico default: `#fff` light / `#11191f` dark) | Page background |
| Secondary (30%) | `--card-background-color` (Pico default: `#fff` / `#1c2731`) | Login `<article>` card, student landing card |
| Accent (10%) | `--primary` (Pico default: `#1095c1`) | Submit button background only |
| Destructive | `--del-color` / `--form-element-invalid-border-color` | Inline error alert border and text only |

**Source:** CLAUDE.md and CONTEXT.md (Pico CSS constraint). No custom hex values are introduced in Phase 2.

Accent reserved for: submit button (`<button type="submit">`) only. All other interactive text (links) use Pico CSS default link color, which is also `--primary` by default — acceptable because links are not styled separately in this phase.

## Pages & Components

### Page: `/login`

**Layout:** Single centered `<article>` card (max-width 360px, centered via `<main class="container">`).

**Elements (top to bottom):**
1. `<hgroup>` — `<h1>SmartAttend</h1>` + `<p>Bitte melden Sie sich an.</p>`
2. `<form method="post" action="/auth/login">` containing:
   - Hidden `<input type="hidden" name="next" value="{{ next }}">` (threads `?next=` param through POST)
   - `<label for="email">E-Mail</label>` + `<input type="email" id="email" name="email" autocomplete="username" required>`
   - `<label for="password">Passwort</label>` + `<input type="password" id="password" name="password" autocomplete="current-password" required>`
   - `<button type="submit">Anmelden</button>`
3. Conditional inline error block (rendered only when `error` variable is truthy):
   - `<small role="alert" style="color: var(--del-color)">{{ error }}</small>` — rendered directly below the submit button, inside the `<form>`

**No JavaScript.** No client-side validation beyond browser-native `required` attribute. Form submits as standard POST; error is re-rendered server-side.

### Page: `/student`

**Layout:** Single centered `<article>` card (max-width 480px, centered via `<main class="container">`).

**Elements:**
1. `<hgroup>` — `<h1>SmartAttend</h1>` + `<p>Anwesenheitserfassung</p>`
2. `<p>Bitte Gerät antippen, um Anwesenheit zu erfassen.</p>` — instruction text
3. No form, no buttons. Placeholder only; Phase 5 replaces content.

### Shared: Base Template (`base.html`)

All pages extend a `base.html` that provides:
- `<!DOCTYPE html>` with `lang="de"`
- `<head>` with `<meta charset="utf-8">`, `<meta name="viewport" content="width=device-width, initial-scale=1">`, Pico CSS `<link>` from `/static/pico.min.css`, and a `{% block title %}` slot
- `<body>` with `<main class="container">` wrapping `{% block content %}`
- No navigation bar (authentication phase — unauthenticated pages have no nav)

## Interaction States

### Login Form

| State | Trigger | Visual |
|-------|---------|--------|
| Default | Page load | Form visible, no error, fields empty |
| Field focused | User clicks/tabs into field | Pico CSS focus ring (`--form-element-active-border-color`) |
| Submitting | User clicks "Anmelden" | Browser-native (no spinner, no JS) |
| Error | Server returns `error` variable | `<small role="alert">` with `var(--del-color)` text appears below submit button |
| Success | JWT set, redirect issued | Browser follows 303 redirect — no UI state needed |

### Protected Route Access (no cookie)

Not a UI component — server issues a 303 redirect to `/login` (from D-07). No error page rendered for this case in Phase 2.

### Wrong Role (valid cookie, wrong role)

Server returns HTTP 403. No custom error template in Phase 2 — FastAPI default error response is acceptable for this internal-only case (D-11: returns 403, no redirect).

## Implementation Notes for Executor

1. Pico CSS file must be downloaded and placed at `app/static/pico.min.css`. Version: latest v2 stable (2.x). Source: `https://cdn.jsdelivr.net/npm/@picocss/pico@2/css/pico.min.css` — download once, commit to repo, serve locally.
2. FastAPI `StaticFiles` must be mounted: `app.mount("/static", StaticFiles(directory="app/static"), name="static")` in `main.py`.
3. Jinja2 `TemplateResponse` requires `Jinja2Templates(directory="app/templates")`.
4. The `?next=` parameter is threaded as follows:
   - GET `/login?next=/checkin?token=<uuid>` → template receives `next` variable
   - Login form includes `<input type="hidden" name="next" value="{{ next | default('') }}">`
   - POST handler reads `next` from form data; only uses it for `student` role (D-10)
5. No JavaScript files are added in Phase 2. No `<script>` tags in any template.
6. `lang="de"` on `<html>` element ensures browser spell-check and autocomplete behave correctly for German.
